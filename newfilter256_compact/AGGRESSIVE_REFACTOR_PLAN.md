# newfilter256_compact 激进内存改造方案（目标 260–320MB 过滤层）

## 1. 目标与约束
- 目标：将 `newfilter256compact` 过滤层从当前约 `~574MB@N=10M` 降到 `260–320MB`。
- 不变约束：
  - 保持 `bin=20, fp=12, k=16, ways=4`。
  - 保持 1/2/3/4 段分支判定语义。
  - 保持可回查 DB（候选 ID 可恢复）。
  - Linux 环境可运行，优先 `AVX512F/BW/VL`，不强依赖 VBMI。

## 2. 当前过滤层内存构成（基线）
- 固定大数组底座（约 423.6MB）：
  - `l1_sizes` + `l1_ids` + `l1_fps` + `l2_heads`（4 路）
- 线性增长部分（约 `~15B/item`）：
  - `seen_*` scratch（epoch/index/count/mask）
  - `l2_next/l2_ids/l2_fps` 预留与扩张

## 3. 激进改造核心策略

### A. L1 槽位全面 bit-packed
- 现状：`id(32bit) + fp(16bit)` 按字节对齐存储。
- 改造：
  - `fp` 压成真实 12bit（不再用 16bit）。
  - `id` 改为 24bit（`N<=16,777,215` 时成立，10M 满足）。
  - 每槽从 6B 逼近 4.5B（理想）。
- 预期收益：L1 主体显著下降（最大头部收益来源）。

### B. 去除“全桶全槽”预分配，改为块式增量分配
- 现状：`bucket_count * k * ways` 全量铺开。
- 改造：
  - L1 改为 segment/block pool（按负载扩展）。
  - 每桶仅保留紧凑索引（起始偏移 + 长度/小元信息）。
  - 热桶连续放置，冷桶不占槽位实体。
- 预期收益：消除大量未使用槽位的固定内存税。

### C. L2 改为紧凑 slab，不使用 per-bucket 链式节点
- 现状：`head + next` 形式，节点扩张碎片化明显。
- 改造：
  - L2 使用 slab 页 + 紧凑偏移索引。
  - 桶内冲突项顺序存储，减少 next 指针和分配器开销。
- 预期收益：降低 L2 元数据比例，提升缓存友好性。

### D. seen/scratch 改为稀疏结构
- 现状：`seen_epoch/index/count/mask` 按逻辑项全量数组。
- 改造：
  - 使用局部哈希缓存（query-local arena）或分块 epoch bitmap。
  - 仅对“候选触达 ID”分配状态，不按全量 N 预开。
- 预期收益：把 `~10B/item` 线性项降到低常数。

### E. 候选去重与分支统计分离存储
- 现状：统计字段与候选路径混杂在主结构。
- 改造：
  - 统计计数独立聚合（thread-local + reduce）。
  - 过滤路径仅保留必要字段，减少热路径对象大小。

## 4. 分阶段落地（建议顺序）

### Phase 1（高收益、可控风险）
1. `id` 改 24bit、`fp` 改 12bit packed  
2. L2 从链式改 slab  
3. 保留现有接口，先让功能与测试全通过  

**阶段目标**：过滤层先降到 `~380–450MB`。

### Phase 2（激进核心）
1. L1 从全预分配改为块式增量分配  
2. seen/scratch 稀疏化  
3. 对候选收集做 SIMD 扫描优化（补偿编码开销）

**阶段目标**：过滤层进入 `~260–320MB`。

### Phase 3（稳定性与性能回归）
1. 与旧实现并跑对照（吞吐/FPR/命中分支）  
2. 大规模 10M 压测 9 轮中位数回归  
3. 输出最终报告与参数冻结  

## 5. 风险点与应对
- 风险 1：bit-packed 增加编码开销，吞吐下降。
  - 应对：热点路径使用 SIMD gather/extract，批量解码。
- 风险 2：24bit ID 限制容量上限。
  - 应对：超过 16.7M 自动切回 32bit 模式（编译期/运行期开关）。
- 风险 3：增量分配引入碎片与并发问题。
  - 应对：固定大小 slab + free-list，避免通用分配器抖动。

## 6. 验收标准（必须同时满足）
- 内存：`N=10M` 过滤层 `<=320MB`（激进目标）；至少 `<=380MB`（保底目标）。
- 准确性：正查命中维持 `~1.0`；负查误命中不劣于当前。
- 语义：1/2/3/4 段分支与“剩余段汉明距离”规则完全一致。
- 工程：`run_linux.sh` 全流程可复现，CSV/PDF 正常产出。

## 7. 对外说明建议
- 将此方案命名为：`newfilter256_compact_v2 (aggressive-memory)`。
- 与当前版本并存，保留回退开关，便于 A/B 测试与论文复现实验。
